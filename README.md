# MiniDrive (Browser-based File Management App)

## 📖 概要
Google Driveのような「ファイル管理」を、サーバーを介さずにブラウザ上だけで完結させるWebアプリケーションです。
未経験からのWebエンジニア転向に向けたポートフォリオとして、**「フレームワークに頼らず、JavaScriptの基礎とデータ構造を理解すること」**を目的に作成しました。

## 🛠 技術スタック
* **Frontend:** HTML5, CSS3, JavaScript (Vanilla ES6+)
* **Database:** IndexedDB (ブラウザ内蔵データベース)
* **Storage:** LocalStorage (簡易的なユーザー管理に使用)
* **Libraries:** JSZip (フォルダ一括ダウンロード用)

## 🚀 こだわったポイント
### 1. サーバーレスでのファイル永続化 (IndexedDB)
通常、画像などのファイル保存はバックエンドサーバーが必要ですが、本アプリでは `IndexedDB` を使用することで、ブラウザ内にGB単位のデータを保存することを実現しました。
* `Blob` オブジェクトを直接DBに格納する処理を実装。
* `Promise` / `async-await` を用いて、非同期であるDB操作を可読性高く記述。

### 2. 直感的なUI/UX
* ドラッグ＆ドロップでのファイルアップロード機能。
* アップロードモードとダウンロードモードのUI切り替え。

## ⚠️ 現在の課題とセキュリティ・リスク分析
本アプリケーションは「学習用プロトタイプ」として開発したため、実務運用を想定した場合、以下の重大なセキュリティ課題があることを認識しています。

### 1. 認証情報の脆弱性 (Critical)
* **現状:** ログインパスワードを `localStorage` に**平文（そのままの文字列）**で保存しています。
* **リスク:** クロスサイトスクリプティング(XSS)等の攻撃を受けた際、パスワードが漏洩し、アカウント乗っ取りのリスクがあります。
* **本来あるべき実装:** * パスワードはサーバーサイドでソルト付きハッシュ化（bcrypt等）を行って保存する。
    * 認証トークン（JWT等）を使用し、HttpOnly Cookieで管理する。

### 2. データ整合性と競合
* **現状:** ユーザー名重複チェック等をクライアントサイドのJavaScriptで行っています。
* **リスク:** サーバーサイドでの排他制御がないため、タイミングによっては整合性が取れない可能性があります。
* **本来あるべき実装:** RDBのユニーク制約やトランザクション処理を用いて整合性を担保する。

## 🏁 今後の展望
現在はJavaScript（フロントエンド）の理解に注力していますが、今後は Node.js やデータベース（SQL）等のバックエンド技術を習得し、上記のセキュリティ課題を解消した「完全なWebアプリケーション」へとアップデートすることを目指しています。
